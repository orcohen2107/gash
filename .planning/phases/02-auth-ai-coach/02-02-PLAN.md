**Tech Stack (Phase 1 verified):**
- @supabase/supabase-js v2.102.1 with auth session management
- expo-secure-store v14.2.4 for platform-secure JWT storage
- Zustand for auth state (no persist middleware — Supabase handles session)
- expo-router v3 for protected routing

**Session Flow (from CLAUDE.md):**
1. User completes OTP verification in Plan 01 → Supabase returns session with JWT token
2. Root layout (`app/_layout.tsx`) subscribes to `supabase.auth.onAuthStateChange`
3. On auth state change: `useAuthStore.setSession(session)` updates Zustand store
4. JWT is automatically stored in expo-secure-store by Supabase client (via ExpoSecureStoreAdapter)
5. On app restart: Supabase client checks SecureStore for JWT → auto-restores session
6. Root layout guard redirects to /auth if no session exists

**Key Constraint (from 02-RESEARCH.md Pitfall 1):**
- Store only JWT in SecureStore (~500B), not full user object (2KB size limit)
- User object lives in Zustand (faster, no size limit)
- Supabase `autoRefreshToken: true` handles token expiry automatically

**Design Decision (from CONTEXT.md):**
- No persist middleware on useAuthStore (Supabase manages session via SecureStore adapter)
- useAuthStore is volatile store — holds current session state only
- Session restored by root layout on every app start via onAuthStateChange listener
</context>
