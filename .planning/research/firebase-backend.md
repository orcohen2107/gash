## Auth — Phone OTP in React Native

### The Right Package: @react-native-firebase/auth

`expo-firebase-recaptcha` is **deprecated and removed as of Expo SDK 48**. Do not use it.
The correct path is `@react-native-firebase/auth` with a custom dev build (not Expo Go).

**Required packages:**
```bash
npx expo install @react-native-firebase/app @react-native-firebase/auth
npx expo install expo-dev-client
```

**app.json plugins:**
```json
{
  "expo": {
    "plugins": [
      "@react-native-firebase/app",
      "@react-native-firebase/auth"
    ]
  }
}
```

### How it works
```typescript
import auth from '@react-native-firebase/auth';

// Step 1: Send OTP
const confirmation = await auth().signInWithPhoneNumber('+972XXXXXXXXX');

// Step 2: Confirm code
await confirmation.confirm(userEnteredCode);
```

### Known Issues and Gotchas

1. **Cannot use Expo Go** — requires a custom dev build (`expo run:ios` / `expo run:android` or EAS Build). This is a hard constraint.

2. **iOS reCAPTCHA redirect with Expo Router** — when reCAPTCHA fires on iOS (simulator or devices without silent push), it redirects to `appScheme://firebaseuth/link`. This can interfere with Expo Router navigation. Workaround: handle the deep link in your root layout, or configure `scheme` in app.json properly.

3. **Android `missing-client-identifier`** — occurs when Play Integrity is not set up. Fix: add SHA-1 and SHA-256 fingerprints to Firebase console for the Android app. Required for production builds.

4. **iOS Simulator** — real device push-based silent verification doesn't work in simulators. Firebase falls back to reCAPTCHA web flow automatically. This is expected behavior, not a bug.

5. **Testing phone numbers** — add test phone numbers in Firebase Console (Authentication > Sign-in method > Phone > Test phone numbers) to avoid SMS costs and reCAPTCHA in development.

**Confidence:** HIGH — verified via rnfirebase.io official docs and GitHub issue tracker.

## Firestore Data Modeling

### Recommended Schema for Gash

The schema uses top-level users collection with subcollections per user. This is correct for this use case: all queries are scoped to a single user, subcollections scale unboundedly (no 1MB document limit), and Firestore Security Rules can enforce user isolation cleanly.

```
/users/{userId}                          ← profile + settings
  /approaches/{approachId}               ← approach logs
  /chatMessages/{messageId}              ← chat history with AI
  /insights/{insightId}                  ← AI-generated insight strings
```

### Document Shapes

**`/users/{userId}`**
```typescript
{
  uid: string;                 // same as Firebase Auth UID
  phone: string;               // E.164 format: +972...
  createdAt: Timestamp;
  totalApproaches: number;     // denormalized counter
  weeklyMission: {
    id: string;
    completedAt: Timestamp | null;
  };
  settings: {
    notificationsEnabled: boolean;
    language: 'he';
  };
}
```

**`/users/{userId}/approaches/{approachId}`**
```typescript
{
  date: Timestamp;             // when it happened
  location: string;            // free text
  type: 'street' | 'bar' | 'gym' | 'event' | 'other';
  opener: string;              // what was said
  response: 'positive' | 'neutral' | 'negative';
  chemistry: number;           // 1-10
  followUp: boolean;           // got contact info?
  notes: string;
  createdAt: Timestamp;
  userId: string;              // denormalized for collection-group queries
}
```

**`/users/{userId}/chatMessages/{messageId}`**
```typescript
{
  role: 'user' | 'assistant';
  content: string;
  createdAt: Timestamp;
  sessionId: string;           // group messages by conversation session
}
```

**`/users/{userId}/insights/{insightId}`**
```typescript
{
  type: 'weekly_summary' | 'pattern' | 'tip';
  content: string;             // Hebrew text generated by Claude
  generatedAt: Timestamp;
  basedOnApproaches: number;   // how many approaches this was derived from
  read: boolean;
}
```

### Query Patterns and Indexes Needed

All queries are scoped to `userId` (the subcollection parent), so they are inherently filtered. The critical compound queries for the Journal/history view are:

| Query | Index Required |
|-------|---------------|
| Filter by `type` + orderBy `date` | Composite: `type ASC, date DESC` |
| Filter by `date` range (date >= X, date <= Y) | Composite: `date ASC` (single field, auto-indexed) |
| Filter by `response` + orderBy `date` | Composite: `response ASC, date DESC` |
| Filter by `followUp == true` + orderBy `date` | Composite: `followUp ASC, date DESC` |

**Rule:** equality filters go first in the index, range/sort fields go last. Firebase CLI will prompt you to create indexes when queries fail in dev — do not skip these prompts.

**Denormalized counter** (`totalApproaches` on the user doc): increment this with a batch write every time an approach is created. Avoids a COUNT() query on the dashboard.

**Confidence:** HIGH — verified via official Firestore data modeling docs and fireship.io patterns.

## Security Rules

Gash's data model is simple: each user owns exactly their own data. The rules below enforce this with zero cross-document lookups (keeping billing clean).

```firestore
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Helper: caller is authenticated and owns this userId path
    function isOwner(userId) {
      return request.auth != null && request.auth.uid == userId;
    }

    // Helper: field is not changed between writes
    function unchanged(field) {
      return request.resource.data[field] == resource.data[field];
    }

    // User profile
    match /users/{userId} {
      allow read: if isOwner(userId);
      allow create: if isOwner(userId)
        && request.resource.data.uid == request.auth.uid;
      allow update: if isOwner(userId)
        // prevent UID and phone from being changed after creation
        && unchanged('uid')
        && unchanged('phone');
      allow delete: if isOwner(userId);

      // Approaches subcollection
      match /approaches/{approachId} {
        allow read, delete: if isOwner(userId);
        allow create: if isOwner(userId)
          && request.resource.data.keys().hasAll([
            'date', 'type', 'opener', 'response', 'chemistry',
            'followUp', 'createdAt', 'userId'
          ])
          && request.resource.data.userId == request.auth.uid
          && request.resource.data.chemistry is int
          && request.resource.data.chemistry >= 1
          && request.resource.data.chemistry <= 10;
        allow update: if isOwner(userId)
          && unchanged('createdAt')
          && unchanged('userId');
      }

      // Chat messages subcollection
      match /chatMessages/{messageId} {
        allow read: if isOwner(userId);
        allow create: if isOwner(userId)
          && request.resource.data.role in ['user', 'assistant'];
        allow update, delete: if false; // messages are immutable
      }

      // AI Insights subcollection
      match /insights/{insightId} {
        allow read: if isOwner(userId);
        allow create: if false;  // only Cloud Functions write insights
        allow update: if isOwner(userId)
          && request.resource.data.keys().hasOnly(['read'])
          && request.resource.data.read == true;
          // users can only mark insights as read, nothing else
        allow delete: if isOwner(userId);
      }
    }
  }
}
```

### Key Design Decisions in These Rules

- **`insights` cannot be created by clients** — only Cloud Functions (which bypass Security Rules via the Admin SDK) write insight documents. This prevents users from injecting fake AI content.
- **`chatMessages` are immutable after creation** — no update/delete prevents message tampering.
- **Schema validation on `approaches` create** — `hasAll()` ensures required fields exist; type and range checks on `chemistry` catch bad client data.
- **No `get()` / `exists()` calls** — all authorization is based on path UIDs. Zero extra reads billed for rule evaluation.

**Confidence:** HIGH — patterns derived from official Firebase security rules documentation.

**Sources:**
- https://firebase.google.com/docs/firestore/security/rules-conditions
- https://oneuptime.com/blog/post/2026-02-17-how-to-write-firestore-security-rules-for-user-based-access-control/view

## Cloud Functions Architecture

### Use Callable Functions, Not HTTP Triggers

For React Native clients, callable functions (`onCall`) are the correct choice. Reasons:

1. **Auth tokens are automatically forwarded** — `request.auth.uid` is available in the function body without any client-side header setup. Firebase SDK handles this transparently.
2. **No CORS configuration needed** — HTTP triggers require CORS handling; callables do not.
3. **Automatic error serialization** — throw `HttpsError` on the server, get a structured error on the client.
4. **App Check integration** — callable functions can enforce App Check tokens automatically.

HTTP triggers (`onRequest`) are better for: webhooks from third-party services, server-to-server calls, or building a REST API consumed by non-Firebase clients. None of these apply to Gash.

### Functions Needed for Gash

```
functions/
  src/
    auth/
      onUserCreate.ts       ← createDocument in /users/{uid} on signup
    approaches/
      onApproachCreate.ts   ← increment totalApproaches counter (Firestore trigger)
    ai/
      sendChatMessage.ts    ← callable: validate + forward to Claude API, save response
      generateInsights.ts   ← callable or scheduled: run weekly analysis against approaches
    admin/
      deleteUserData.ts     ← callable: GDPR delete all subcollections for a user
```

### Callable Function Pattern

```typescript
import { onCall, HttpsError } from 'firebase-functions/v2/https';
import { getFirestore } from 'firebase-admin/firestore';

export const sendChatMessage = onCall(
  { region: 'europe-west1' },  // closer to Israel than us-central1
  async (request) => {
    if (!request.auth) {
      throw new HttpsError('unauthenticated', 'Must be signed in.');
    }

    const { message, sessionId } = request.data;
    const userId = request.auth.uid;

    // call Claude API, save to Firestore, return response
    // ...
  }
);
```

### Runtime Recommendation

Use **Cloud Functions v2** (`firebase-functions/v2`). v1 is legacy. v2 supports:
- Min/max instance configuration (prevent cold starts on free tier)
- Concurrency per instance (more efficient)
- Secrets Manager integration for Claude API key (safer than environment variables)

**Region:** Deploy to `europe-west1` (Belgium). This is geographically closest to Israel of all Firebase Function regions and will minimize latency for Israeli users compared to `us-central1`.

**Confidence:** HIGH — verified via official Firebase callable functions docs and rnfirebase.io.

- 20 reads (load context) x 30 days = 600 reads

- Reads: ~1,200
- Writes: ~650

- Reads: 1,200 / 100,000 x $0.06 = $0.0007

- 40+ daily active users before hitting read limits

**Practical conclusion:** Firestore costs are negligible until the app reaches 1,000+ DAUs. The dominant cost drivers will be Cloud Functions invocations and the Claude API (which are separate billing). Budget planning should focus on Claude API costs, not Firestore.

## Offline Support

### Status: Works Well with react-native-firebase

When using `@react-native-firebase/firestore` (the native SDK, not the JS SDK), **offline persistence is enabled by default** on both iOS and Android. No configuration required.

```typescript
// Only needed if you want to DISABLE it (e.g., for sensitive data screens)
import firestore from '@react-native-firebase/firestore';

await firestore().settings({ persistence: false });
// Must be called before ANY Firestore operation
```

### What Works Offline
- Reading previously fetched documents (served from local cache)
- Writing documents (queued locally, synced when connection restored)
- Real-time listeners (receive cached data immediately, update when online)

### What Does NOT Work Offline
- **Transactions** — will fail. Do not use transactions for critical user flows.
- **First-time reads** — no cache yet, data unavailable until connected.
- **Server timestamps** — `FieldValue.serverTimestamp()` will be approximate until sync.

### Gotcha: Avoid `await` on Write Operations in Offline Mode

```typescript
// BAD: blocks UI until server confirms write (hangs when offline)
await firestore().collection('approaches').add(data);

// GOOD: fire and forget; Firestore queues offline, UI unblocked
firestore().collection('approaches').add(data);
```

For Gash specifically, approach logging should use fire-and-forget writes so the UI feels instant on poor connections (common on Israeli mobile networks in the field).

### Firebase JS SDK vs react-native-firebase for Offline

If you use the **Firebase JS SDK** (not the native SDK), offline persistence does NOT work reliably in React Native. This is a known limitation — the JS SDK uses browser APIs (IndexedDB) not available in React Native's environment. This is another strong reason to use `@react-native-firebase` over the JS SDK for this project.

## Key Recommendations

- **Use `@react-native-firebase` throughout** — not the Firebase JS SDK. It gives you native offline persistence, proper phone auth, and better performance. The JS SDK is for web; rnfirebase is for React Native.

- **Build a custom dev client immediately** — `expo-dev-client` + `eas build --profile development`. You cannot test phone auth or any `@react-native-firebase` feature with Expo Go. Do this in Phase 1 before any auth work.

- **Deploy Cloud Functions to `europe-west1`** — not the default `us-central1`. Belgium is the closest Firebase region to Israel. Latency matters for AI chat response time.

- **Use callable functions for all client-facing functions** — automatic auth forwarding, no CORS, structured error handling. Reserve `onRequest` for webhooks only.

- **Store Claude API key in Firebase Secrets Manager** — not in environment variables. Use `defineSecret('CLAUDE_API_KEY')` in v2 functions syntax.

- **Add test phone numbers in Firebase Console** — avoids SMS costs and reCAPTCHA friction during development. Use `+972 555 000001` style numbers with fixed codes like `123456`.

- **Configure Android SHA fingerprints before any device testing** — without them, phone auth throws `missing-client-identifier` on Android. Get fingerprints from `eas credentials` or local keystore.

- **Use fire-and-forget writes for approach logging** — do not `await` Firestore writes in the UI flow. Queue them and let Firestore handle sync. This makes the app feel fast on poor mobile connections.

- **Composite indexes are required for Journal filters** — create them proactively in `firestore.indexes.json` rather than waiting for runtime errors. Needed indexes: `(type, date)`, `(response, date)`, `(followUp, date)`.

- **Denormalize `totalApproaches` counter on the user document** — increment it in an `onApproachCreate` Firestore trigger function. Never COUNT() subcollections on the client for dashboard metrics.

- **Security rules: block client writes to `insights`** — only Cloud Functions (Admin SDK) should write AI-generated content. Clients can only mark insights as `read: true`.

- **Firebase is the right choice over Supabase for this project** — the app is mobile-first, uses phone OTP auth, needs real-time sync, and has a simple schema. Firebase's native mobile SDKs, built-in offline persistence, and zero-infra serverless functions are a better fit than Supabase's PostgreSQL + PostgREST for a solo-founder MVP. Switch to Supabase if v2 requires complex relational queries or a web dashboard.
